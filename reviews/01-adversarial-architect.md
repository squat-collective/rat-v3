# Adversarial architect review — RAT v3

*Reviewer role: adversarial architect. Mandate: assume this ships in 12 months exactly as designed, with modest adoption (~50 plugins, ~100 deployments). At year 3, what looks wrong?*

## Headline

The architecture is **internally consistent and unusually disciplined** — but it bets the whole platform on an assumption it never tests: that the 16 axes are *orthogonal*. They aren't. The data-plane axes are a tightly-coupled cluster that share a data representation, a credential flow, and an isolation boundary the policy-free core cannot mediate. The thing most likely to look wrong in 3 years is that "everything is an independent plugin" was true for the stateless axes (UI, notifications, observability) and a costly fiction for the stateful, data-touching ones.

## What's strong (don't skip this)

1. **The engine/runtime/format/strategy split is a genuine insight.** Naming Arrow as the seam between "SQL→Arrow" (engine) and "Arrow→Arrow ops" (runtime), and making strategies consume the runtime rather than the engine, is the cleanest part of the whole design. It's a real improvement over v2 and it's load-bearing in the right way.
2. **Data plane bypasses the core for bytes.** (overview.md, "Communication model.") This is correct and non-obvious. Most platforms that start as a "thin coordinator" end up a chokepoint because bytes flow through them. Explicitly forbidding that is the single best decision in the doc set.
3. **Reconciliation over orchestration, with events-as-hints.** The inbox note "the reconciler is the source of truth, not the events… events are hints" (ideas/inbox.md, event-bus failure modes) is exactly the right K8s lesson. Idempotent re-read on every loop is what makes the async bus survivable.
4. **ADR discipline is above the bar for greenfield work.** Rejected-alternatives tables, honest `Consequences` sections, "track the temptation count" as a drift metric. This is how you keep a project honest. Most teams discover this discipline at year 4, in pain.
5. **"No exotic bets" on the stack (ADR-002).** Go, NATS, JSON Schema, Apache 2.0, K8s patterns. Every choice is a boring, proven component. The risk budget is spent entirely on the *architecture*, not the stack. Correct allocation.

## Findings — what will break

### Finding 1: The reconciler *is* a scheduler, and it lives in the core — so the core will creep
**Citation:** overview.md, "The six core things" #5 (Reconciler) vs. plugin axes (`scheduler-backend`). Reconciler loop pseudocode: *"If scheduled to run now: check plane is healthy… Emit `pipeline_run_requested`."*
**Claim:** Scheduling is declared a plugin axis, but the reconciler — which is core — already contains scheduling logic ("if scheduled to run now"). The boundary between "scheduler-backend plugin" and "reconciler in core" is undefined, and every hard scheduling feature (dependency ordering between pipelines, backfills, retry-with-policy, concurrency limits, catch-up after downtime) wants to live next to the reconciler, i.e. in the core.
**Why it breaks:** This is precisely how Kubernetes' own core grew. etcd stayed ~50k LOC; the *controller-manager and scheduler* — the analog of RAT's reconciler — are now enormous. The doc compares core size to etcd/NATS (overview.md line 59), but RAT is building the controller-manager, not etcd. The "5-10k LOC, stays six things" budget is benchmarked against the wrong component. Expect the first three "this can't be a plugin" ADR exceptions to all cluster around reconciler/scheduling semantics.
**Probability:** likely-fails (the 5-10k budget will be breached within 18 months; the *principle* survives but the LOC claim won't).
**Fix direction:** Define now, in a contract, the exact line between reconciler (detects drift, emits intent) and scheduler-backend (decides *when*); write the ADR for it before Phase 1, not after the creep.

### Finding 2: The 16 axes are not orthogonal — the data-plane five are a coupled cluster
**Citation:** ADR-001, "Why this specific decomposition" (axes presented as independent); overview.md data-plane list (engine, runtime, format, strategy, catalog, storage).
**Claim:** engine → runtime → format → strategy → catalog → storage are not independent axes; they are stages of one coupled pipeline that share Arrow buffers, file paths, and vended credentials. A new `format` plugin must actually interoperate with every engine's Arrow dialect, every catalog's snapshot semantics, and every storage's credential model. The manifest's `requires`/`suggests` (plugin.yaml) expresses *structural* dependency ("I need a format with capability=merge") but cannot express *semantic* compatibility ("my merge works against DuckDB's Arrow output but not ClickHouse's").
**Why it breaks:** This is the OSGi outcome the docs explicitly fear (competitors.md line 78: "the OSGi outcome — technically great, ecosystem stillborn"). With 50 community plugins, the compatibility matrix is combinatorial (engines × runtimes × formats × catalogs × storages). Nobody tests the full matrix; users hit "plugin A + plugin B silently corrupts data" and the platform gets a reputation for fragility. The manifest validates that the pieces *fit*, not that they *work together*.
**Probability:** likely-fails (this is the central technical risk; it's the difference between "K8s for data" and "another OSGi").
**Fix direction:** Add a conformance-test-suite contract per axis (a plugin isn't "valid" until it passes the reference golden-data suite against the reference peers) and publish a verified-compatibility matrix — treat semantic compat as a first-class artifact, not an emergent hope.

### Finding 3: Multi-tenant isolation is an emergent property of four independently-authored plugins
**Citation:** vision.md line 29: *"Compliance is a configuration, not a fork."* overview.md: `storage.VendCredentials(prefix)`; tenancy/identity/billing all plugins; "data plane bypasses core."
**Claim:** For tenant A to be unable to read tenant B's bytes, the tenancy plugin, the identity plugin, the storage plugin (credential vending), and the engine plugin must *all* agree on the same boundary — but the core, the only component they share, explicitly owns no policy and never sees the bytes. Isolation is therefore an emergent property of four separately-authored, separately-versioned plugins, mediated by a core designed to stay out of the way.
**Why it breaks:** Cross-tenant data leaks come from the *gap between* components, and here every gap is a plugin boundary with no central enforcement point. "Data plane bypasses the core" (a strength for performance) means the core *cannot* be the policy enforcement point for data access even if it wanted to. A mis-scoped `VendCredentials` in one storage plugin defeats a correct tenancy plugin. "Compliance is a configuration" is the most dangerous sentence in the docs: it markets a property the architecture makes *harder* to guarantee, not easier.
**Probability:** likely-fails at the enterprise/multi-tenant topology (topology D); edge-case-only for solo/team.
**Fix direction:** Make credential vending and tenant-scoping a *core-mediated* path (the one policy exception), or define a mandatory isolation-contract that storage+identity+tenancy plugins must jointly satisfy and that the core can audit — and stop selling multi-tenant SaaS as "just three plugins."

### Finding 4: "Major version only" capability versioning forces ecosystem-wide flag days
**Citation:** ADR-002 D4; research/prior-art/README.md lists "npm peerDependencies — peer deps are about negotiation, not strict resolution."
**Claim:** Capabilities are `rat://kind/v1/cap`, major-only, no ranges (D4). When `engine/v1` → `engine/v2` ships a breaking change, every runtime/format/strategy/catalog plugin that talks to engines must cut over simultaneously or maintain parallel v1+v2 implementations. There is no gradual-migration mechanism because there are no ranges and no negotiation.
**Why it breaks:** K8s gets away with major-only because it runs a multi-year deprecation treadmill staffed by a large paid org and controls every first-party API. A 50-plugin community with multiple authors has no coordinator to run a flag day. The docs cite npm peerDependencies as prior art *for negotiation* and then choose the opposite (strict, no negotiation). You're reproducing the exact rigidity that semver ranges were invented to solve, in an ecosystem with less central coordination than npm.
**Probability:** might-fail (survives until the first breaking change to a popular capability; then it bites hard and is hard to retrofit).
**Fix direction:** Keep major-in-URI for routing, but add a minor/feature-capability negotiation field in the manifest (`requires: engine/v1 with capabilities [x,y]`) so a plugin can degrade gracefully instead of breaking — negotiation, per the prior art you already cited.

### Finding 5: The state gateway's KV abstraction leaks consistency semantics into core correctness
**Citation:** overview.md core #3 (State gateway: `Get/Put/Watch/List`); state-backend plugins = postgres, sqlite, dynamodb, etcd, in-mem; ADR-002 D5 (reconciler uses "state-backend's CAS primitive").
**Claim:** The state gateway is a KV interface, but the reconciler's correctness depends on consistency guarantees that differ wildly across the listed backends: sqlite gives serializable transactions; DynamoDB gives per-item CAS but no cross-key transactions; in-mem gives nothing durable. The reconciler logic is *one* codebase but must be correct against *all* of them. So either the core is written to the weakest backend (no multi-key transactions — over-constrained on sqlite/postgres) or it assumes transactions and is silently wrong on DynamoDB.
**Why it breaks:** "Same core, different plugins, core never changes" (overview.md, Scalability) is false here: the consistency model is not pluggable without leaking into core logic. Leader-election-via-CAS (D5) works on all of them, but the *reconciliation of related objects* (plane→pipeline→run→branch) needs more than single-key CAS. This is the axis where the "state is just a plugin" abstraction is thinnest.
**Probability:** might-fail (works fine for the sqlite/postgres default; surfaces as subtle race/corruption when someone actually deploys the DynamoDB backend the docs advertise).
**Fix direction:** Specify the *minimum* consistency contract the state gateway requires (e.g. "single-key linearizable CAS + watch, no multi-key txn") and build the reconciler to that floor — then loudly mark backends that can't meet it as unsupported, rather than listing DynamoDB as a peer of postgres.

### Finding 6: There is a hidden "tier 0" of plugins that are really core
**Citation:** ADR-001, core #3 (State gateway "interface is in-core because the registry itself needs somewhere to persist — but the implementation *is* a plugin"); deployment-runtime listed as an ordinary plugin axis.
**Claim:** The chicken-and-egg argument used to justify the six core things applies equally to two declared "plugins": **state-backend** (the registry needs it before any plugin loads) and **deployment-runtime** (you need it to *run* any out-of-process plugin — but it is itself a plugin that must be run). The doc admits the state *interface* is core but calls the implementation a plugin; for deployment-runtime it doesn't even do that. There is a bootstrap tier of plugins that cannot be swapped at runtime like the others and are effectively part of the core's trusted base.
**Why it breaks:** "Six things" undercounts. In practice the trusted, bootstrap-critical surface is: six core things + state-backend + deployment-runtime + the embedded event bus (NATS is named in D2 as embedded-in-core, so the "bus transport is a plugin" claim is already half-false for the default). That's closer to *nine* irreducible things. The discipline metric ("the core stays six") is measuring the wrong number, and the marketing claim "swap any plugin at runtime" is false for the bootstrap tier.
**Probability:** edge-case-only as a *bug*, but likely-fails as a *truth-in-advertising* problem — it erodes the central "everything is a plugin" claim under scrutiny.
**Fix direction:** Name the bootstrap tier explicitly. Define "tier-0 plugins" (state-backend, deployment-runtime, bus) that are selected at boot and not hot-swappable, and stop claiming the embedded NATS bus is "just a pluggable transport" when it's compiled in.

### Finding 7: "Swap plugins one at a time, no migration" is false for every stateful axis
**Citation:** vision.md line 23: *"swap plugins one at a time — same binary, same conversation, no migration."*
**Claim:** This is true for stateless axes (UI, notifications, observability, identity) and false for the stateful ones. Swapping state-backend sqlite→postgres is a data migration. Swapping format Iceberg→Delta is a full table rewrite. Swapping catalog Nessie→Unity is a metadata migration. Swapping storage S3→GCS is a byte copy. The "no migration" promise conflates two very different kinds of axis.
**Why it breaks:** Early adopters take the promise literally, try to swap their format or catalog, and discover the platform has no migration story (D7 explicitly punts all migration tooling). The gap between marketed pluggability and real pluggability becomes the platform's credibility problem — exactly the "extensible-but-not-actually-used" failure mode named in competitors.md.
**Probability:** likely-fails as an expectations problem; the architecture is fine, the *claim* is not.
**Fix direction:** Split the axis taxonomy in the docs into "hot-swappable (stateless)" vs "migration-required (stateful)" and ship at least one reference migration path (sqlite→postgres) so the stateful-swap story isn't vaporware.

### Finding 8: The contract triple is frozen in Phase 0, before any plugin exists to stress it — and the AI-rewrite hatch doesn't cover it
**Citation:** ADR-001 Migration "Phase 0… Lock the contract triple"; CLAUDE.md "Core last: implement the core only after enough plugins exist"; ADR-002 D1 meta-principle ("AI-assisted rewriting lowers the cost of language migration").
**Claim:** The sequencing freezes the *most expensive-to-change* artifact (the proto + manifest + capability contract that 50 plugins will depend on) earliest, when zero real plugins have stressed it, and defers the *cheapest-to-change* artifact (the core, which one person can rewrite). The AI-rewrite escape hatch is repeatedly invoked to de-risk the language choice — but you can AI-rewrite a 10k-LOC core; you cannot AI-rewrite a 50-author ecosystem that hard-coded `engine/v1`.
**Why it breaks:** Contracts designed in a vacuum are wrong in ways only revealed by the second and third real implementation of each axis. By the time those exist, the contract is published and depended-upon. The project optimized reversibility for the thing that's already cheap to change and locked the thing that's permanent. The open thread "performance benchmarks before we lock the contract… if it's >10ms per format.Resolve the whole model is in trouble" (vision conversation, open threads) admits the contract is being locked before the one measurement that could invalidate it.
**Probability:** likely-fails (some axis contract will need a breaking revision after real plugins exist, and there'll be no cheap way to do it).
**Fix direction:** Build *two* independent reference implementations per critical axis (engine, format, catalog, state) *before* declaring the contract v1-frozen, and run the per-RPC benchmark spike first — let real second-implementations and real latency numbers be the forcing function, not a paper review.

## Missed questions

The ADRs don't ask these, and should:

- **What is the per-pipeline RPC amplification, and at what file/partition count does the contract tax dominate?** A pipeline over a 10k-partition table does how many `format.Resolve` / `catalog.GetTable` / `storage.VendCredentials` hops? In-process (solo) it's free; cross-container (team+) it's network. The contract is the same; the performance is not. This is unmeasured and the contract is being frozen anyway.
- **Who owns the compatibility matrix?** When 50 plugins exist, what tells a user that `engine-clickhouse` + `format-delta` + `catalog-unity` is a tested combination vs. a data-corruption landmine? (See Finding 2.)
- **What is the conformance bar for "this is a valid `format` plugin"?** Manifest validity ≠ behavioral correctness. Without a golden-data conformance suite per axis, "valid plugin" means "well-formed YAML," not "writes correct Iceberg."
- **How does a tenant boundary survive a malicious or buggy plugin?** Sandboxing is deferred to Q15, but the entire value prop is "install lots of third-party plugins." You're deferring the security of the exact boundary the thesis leans on hardest.
- **What's the deprecation/migration treadmill for a breaking capability bump, given there's no central org to run a flag day?** (See Finding 4.)
- **Does "core never sees data" preclude core-enforced data governance** (column-level access, PII masking, audit-of-actual-reads) that enterprises require — and which the bypass-the-core design structurally cannot provide?

## The single most dangerous bet

**That the contract triple can be designed correctly in Phase 0, frozen, and then have an orthogonal-plugin ecosystem grow on top of it — when the data-plane axes are not actually orthogonal.**

This is the bet to revisit in 12 months with real evidence. Findings 2 and 8 are the same bet seen from two sides: the axes are sold as independent (so a community can build them in parallel against a frozen contract), but the data-plane axes are a coupled pipeline sharing Arrow, paths, credentials, and isolation semantics that the contract-as-specified cannot fully capture. Every other finding is patchable — the reconciler can be re-scoped, the versioning scheme extended, the migration tooling added later. But if the contract is frozen early *and* the orthogonality assumption is false, you've published a coupling-blind contract that 50 plugins now depend on, and the only fix is the breaking change the whole "major-version-only, AI-rewrite-the-core" strategy is least equipped to absorb.

The tell is in the docs' own words: the project's stated failure mode is *"the OSGi outcome — technically great, ecosystem stillborn"* (competitors.md). The mechanism by which that happens is not a stillborn ecosystem of indifference; it's an ecosystem where plugins technically conform and don't actually work together. That is the default trajectory of this design unless semantic compatibility (conformance suites + verified matrix) is made a first-class, contract-level concern *before* Phase 0 freezes anything.

If I could force one change: do not freeze any data-plane contract until two independent real implementations of that axis exist and have been run against each other on golden data. Let the second implementation, not the first review, be what declares the contract done.
