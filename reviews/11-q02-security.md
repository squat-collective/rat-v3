# Q02 external review (SIMULATED dry-run) — security
Reviewer persona: control-plane threat-modeler — container isolation (runc/gVisor), STS credential vending, capability authz, supply-chain integrity (Sigstore/in-toto)  ·  Lens: security  ·  Date: 2026-06-02  ·  Provenance: SIMULATED (AI persona)

## Bottom line

The control-plane boundary is in genuinely good shape for a spike: C5 authz is *derived* from manifests (not hardcoded), every decision is audited (C4), conformance is *verified* not declared (D4 / `NewVerified`), and the podman runtime *actually* enforces a kernel-level isolation floor (D1) instead of self-attesting it. Review 04's headline ("every security control is optional") is largely answered — the core now enforces, not delegates. **But I would not run this near multi-tenant data yet**, and the reason is structural, not a bug: the **data plane bypasses the core, so the platform's strongest control (the core's mediation) does not exist on the bytes leg** — and the *planned* C2 channel-auth fix, being a core fix, cannot reach it. The single thing to fix first: **make data-plane identity a contract obligation**, not a per-producer courtesy. The Arrow ticket binds `{caller,tenant}`, but every third-party, polyglot producer independently decides whether to *believe* the presenting identity — and the reference believes raw HTTP headers (`bulkleg_test.go:39`). The whole "install many third-party plugins" bet puts the most security-critical code (the byte gate) in the least-trusted place (the third party), with the core unable to backstop it. C2 closes the control-plane identity gap cleanly; it does nothing for the data plane.

## Findings

### F1 — Data-plane identity is self-asserted at the producer; C2 cannot reach the bytes leg
- **Severity:** High (Critical for an untrusted-producer multi-tenant deployment)
- **Area:** data-plane
- **Finding:** The Arrow ticket is bound to `{stream,caller,tenant}` and validated correctly (`core/arrowticket/arrowticket.go:70-92`). But because the bytes leg *bypasses the core entirely* (by design — vision §"data plane bypasses control plane"), the *presenting* identity that `Validate(ticket, streamID, caller, tenant)` is checked against comes from the producer's own transport, not the core. The reference producer takes it from unauthenticated request headers: `m.Validate(ticket, bulkStreamID, r.Header.Get("X-RAT-Caller"), r.Header.Get("X-RAT-Tenant"))` (`core/arrowticket/bulkleg_test.go:39`). The frozen contract only says a producer "MUST issue tickets that are … bound to {caller_plugin, tenant, this stream}" (`common/v1/data.proto:56-63`) — it obligates the *minting* but never obligates the producer to *authenticate the channel it checks the binding against*. So the binding check reduces to "the caller told me it's tenantB," which is exactly the spoof C2 is meant to kill on the control plane.
- **Why it matters:** A leaked or guessed ticket presented from any connection succeeds if the attacker simply sets `X-RAT-Caller`/`X-RAT-Tenant` to the bound values — the "cross-tenant refused" property in `bulkleg_test.go:109-112` holds *only because the test honestly sets the wrong header*; a real attacker sets the right one. The cross-tenant byte-access defense is therefore honor-system on the third party's transport. C2 (per-plugin token + mTLS, re-derived in `gateway.openCall`, `gateway.go:201-204`) is a *core* fix; the bytes leg has no core in it, so C2 structurally cannot protect it. This is the confused-deputy / cross-tenant case the brief's boundary #4 worried about, and it survives the one fix everyone is counting on.
- **Freeze-reopen?** **yes (soft — contract-obligation, not wire-message).** The `bytes ticket` field is opaque and sufficient, but the *frozen contract surface* is missing a normative MUST: `data.proto` SEC-14 / the ArrowStream producer conformance must add "the producer MUST verify the presenting channel's authenticated identity equals the ticket's bound `{caller,tenant}`; transport/app headers are not sufficient," with a conformance vector for a *wrong-channel-right-header* attempt. This is best added before the freeze ships externally — a conformance MUST is part of the frozen contract, even though no message shape changes.
- **Cross-lens:** **Maintainer CONCEDED in full** — "no frozen contract MUST anywhere obliges a polyglot third-party producer to authenticate the presenting channel; it's an honor-system property — exactly the shape ADR-005 rejected for control calls and knowingly accepted for the bytes leg (freeze residual R2). Backlog E8 is adjacent but does not state this as a per-producer Flight-channel obligation. Net-new finding as a contract-obligation gap." Rated it the highest of my three. **Architect cross-lens resolved:** C2 splits by plane — the control-plane fix is a clean *patch* (wire unchanged; `gateway.openCall` swaps the identity *source*, providers already authenticate the core peer), but the bytes leg, having no core peer, is where the same "integrity rests on transport auth" commitment degrades to honor-system → this is the freeze-reopen item, filed here not against the control-plane model.
- **Suggested direction:** Make the bytes-leg identity contract-mandatory: producers MUST run Arrow Flight over mTLS (or a token the *core vended alongside the ticket*) and bind the check to the authenticated peer, never to app-layer headers. Ship it as the SDK default so third parties can't get it wrong by omission. Consider minting the ticket to a *channel/cert fingerprint* so a ticket is unusable off the connection it was issued for.

### F2 — Arrow ticket: in-memory single-use + a single un-rotatable symmetric key
- **Severity:** High
- **Area:** data-plane
- **Finding:** Single-use is enforced by an in-process map: `used map[string]bool` on the `Minter` (`core/arrowticket/arrowticket.go:54,84-90`). It is per-producer-process and non-durable. The package doc concedes "single-use is producer-enforced in-memory." Separately, `NewMinter(key []byte)` (`:58`) takes **one symmetric HMAC key with no key id** — contrast `conformance.Attestation`, which carries a `KeyID` for rotation/agility (`core/conformance/attestation.go:31,80-84`). The ticket has no rotation identifier and no per-producer key dimension in the design.
- **Why it matters:** (a) **Replay reopens on restart/scale-out.** A producer crash-and-restart or a second replica starts with an empty `used` map, so a captured-but-"already-used" ticket replays within its TTL — "single-use" silently becomes "single-use per process lifetime per replica." Any horizontally-scaled or auto-healed producer (the reconciler *restarts* unhealthy plugins) breaks the invariant. (b) **Key blast radius.** If the HMAC key is shared across producers, compromising *one* third-party producer process yields the key to forge tickets for *any* `{stream,caller,tenant}` on *any* producer sharing it; with no key_id there's no clean rotation — you must flag-day every producer. There is no central audit of byte fetches (the core is bypassed), so neither failure is observable from the control plane — **SRE co-signs this as undetectable**: a replayed DoGet leaves no central audit record, so this is not just exploitable but an incident with no signal (their F2/F7 cross-lens concurs).
- **Freeze-reopen?** no (ticket is opaque bytes) — key-management + a shared/distributed single-use store are enforcement-layer.
- **Cross-lens:** **Maintainer CONCEDED all three** — no key_id/rotation (confirmed the `conformance.Attestation.KeyID` contrast is "the right tell"); per-Minter key = per-producer-process with no defined sharing model (a ticket is only verifiable by the producer that minted it — "undesigned for any pooled/replicated producer"); in-memory single-use → restart re-opens the replay window and replicas don't share the used-set; "the only thing bounding damage today is short TTL." Designed fix (shared single-use store + key_id + rotation) is "not designed." key_id absence is net-new beyond the replica-replay caveat maintainer had pre-flagged.
- **Suggested direction:** Per-producer keys with a `key_id` inside the ticket (mirror the conformance/audit keyring); shared single-use store (the state-backend CAS the leader-lease already uses) or make tickets *stateless-single-use* by binding to a nonce the consumer must echo over an authenticated channel; document that in-memory single-use is non-durable and unsafe for replicated producers.

### F3 — Seccomp is advertised in the I9 profile but is NOT part of the enforced minimum; "unconfined" is accepted
- **Severity:** Medium-High
- **Area:** core
- **Finding:** `checkI9Minimum` enforces only `run_as_non_root ∧ drop_all_capabilities ∧ no_new_privileges` (`core/deploymentruntime/isolation.go:17-27`). Seccomp is in the frozen profile (`seccomp_profile`, `deploymentruntime/v1` field 6) and the contract says the default "must be at least RuntimeDefault-equivalent" — but the gate never checks it. The podman runtime then passes a non-empty profile straight through: `if s := iso.GetSeccompProfile(); s != "" && s != "RuntimeDefault" { args = append(args, "--security-opt=seccomp="+s) }` (`core/deploymentruntime/podman.go:111-113`). A `LaunchSpec` with `seccomp_profile: "unconfined"` therefore yields `--security-opt=seccomp=unconfined` — seccomp **disabled** — and `checkI9Minimum` allows it. The isolation receipt would honestly report `seccomp_profile: "unconfined"` (`:140`), but nothing *refuses* the downgrade.
- **Why it matters:** The unfiltered syscall surface is precisely what container-escape CVEs need (CVE-2022-0492 cgroups `release_agent`, keyctl/`unshare`-class primitives). The brief asks "can a plugin influence its own LaunchSpec to request weaker isolation?" — for the three booleans, no (podman hardcodes them regardless of spec, good). For seccomp, **yes**: whoever authors the desired state (and per review 04 TS-5 a plugin that can write desired state is in scope) can request `unconfined` and the "minimum" floor waves it through while the system still presents as I9-compliant.
- **Freeze-reopen?** no — `checkI9Minimum` is core enforcement, not wire. Fix is a one-line gate addition.
- **Cross-lens:** none yet (offered to sre as part of the DoS/isolation thread).
- **Suggested direction:** Add seccomp to the minimum: refuse `unconfined` and any profile weaker than RuntimeDefault; allowlist named profiles; treat empty as RuntimeDefault (already does). Better, make the runtime *impose* the max rather than *honor* a caller-supplied weaker value.

### F4 — No resource-limit / pids enforcement: the I9 floor contains privilege but not consumption (DoS / noisy-neighbor)
- **Severity:** High
- **Area:** core / operability
- **Finding:** The frozen `LaunchSpec` carries `ResourceQuantities limits` (cpu/memory) (`deploymentruntime/v1` fields 68-69), but the podman runtime never applies them — there is **no** `--memory`, `--cpus`, or `--pids-limit` in the launch args (`core/deploymentruntime/podman.go:73-117`, verified by grep: none). `checkI9Minimum` says nothing about resources either. There is no `pids-limit` field in the proto at all (fork-bomb has no contract dimension).
- **Why it matters:** I9 stops a plugin from *escalating* (non-root, cap-drop, no-new-privs) but not from *exhausting the host*. A hostile or merely buggy third-party plugin fork-bombs (`--pids-limit` absent), balloons memory (OOM-killer reaps neighbors), or pins CPU — on a shared host that is a cross-tenant **availability** breach and a classic noisy-neighbor. The reconciler will faithfully *restart* the offender, turning a DoS into a crash-loop amplifier. The contract already has the cpu/memory knobs; the enforcing runtime drops them on the floor, so the spike's "full I9 profile" is privilege-isolated but not resource-isolated.
- **Freeze-reopen?** no for cpu/memory (the proto fields exist; this is an implementation gap) — a `pids-limit`/`ulimit` field is a small additive proto extension, not a break.
- **Cross-lens:** **SRE concurred and raised the control-plane half** — the single-leader reconciler holds one global mutex and makes *unbounded* (no per-call deadline) Healthcheck/Launch/Terminate RPCs to the runtime while holding it (`reconciler.go:97-98`), so a single hung/slow plugin stalls convergence + Status reads for **all** tenants. Together with this resource-limit gap that is one multi-tenant availability story: a tenant's plugin can both (a) exhaust the shared host unboundedly *and* (b) freeze the shared leader loop. The **data-plane bypass does NOT contain this** — the stall and the exhaustion are control-plane / host-level, which every tenant shares. We agreed to jointly frame it as "availability is a tenancy boundary the spike does not isolate" (SRE files the reconcile-fairness half as primary High; this is the resource-isolation half).
- **Suggested direction:** Plumb `limits.cpu`/`limits.memory` into `--cpus`/`--memory`; add a `pids-limit` default; make a resource floor part of the I9 minimum for untrusted plugins; bound the reconciler's runtime RPCs with per-call deadlines so one plugin can't pin the leader. Consider this a D1 completeness item, not GA.

### F5 — Conformance trust root: single authority, no revocation, no attestation expiry ("conformed forever")
- **Severity:** Medium
- **Area:** ecosystem / supply-chain
- **Finding:** D4 verification is sound as far as it goes — `NewVerified` refuses unattested, partially-attested, or badly-signed providers (`core/registry/verified.go:33-46`), and the signature commits to the `KeyID` (defeating key-substitution, `attestation.go:38-41`). But: (a) `Attestation` has **no expiry** and **no revocation** — `Conforms` is a static set membership (`attestation.go:45-52`); a plugin conformed once is conformed forever, even after a CVE is disclosed against that exact version. (b) The `Authority` is a single keyring (`attestation.go:67-73`); revoking a compromised key by removing its `KeyID` invalidates **every** plugin attested under it (availability cliff), and there is no per-attestation revocation in between. (c) A single signing authority is a single point of forge: compromise it and you mint conformance for `rat-format-evil`, which then sails through `NewVerified` and becomes a trusted provider.
- **Why it matters:** This is the supply-chain boundary the whole marketplace bet rests on. "Capability negotiation is a lie unless conformance is enforced" (review 02) is now half-solved: conformance *is* enforced, but its *freshness and revocability* are not. A conformed-then-vulnerable plugin keeps its trust badge indefinitely; a key compromise has no graceful recovery. Sigstore/in-toto ecosystems learned this: attestations need expiry + a transparency/revocation channel, not just a signature.
- **Freeze-reopen?** **yes (soft)** — adding an `expires_at` and a revocation reference to the attestation / `marketplace.proto conformed_capabilities` is a wire-shape change best made before freeze ships externally; doing it post-freeze without a major is awkward.
- **Cross-lens:** **Maintainer CONCEDED on every point** — `Authority` is a single keyID→pubkey map; `Verify` checks only "signature valid + keyID known" (`attestation.go:79-87`); no expiry field on `Attestation`, no revocation list; the keyring *can* hold multiple keyIDs but there's no notion of *which authority may attest what*, so "a single forged/leaked trusted key = full trust of arbitrary `provides`," and "a conformed-then-CVE'd plugin stays trusted forever." The spike deliberately proves the *mechanism* (signed `declared == conformed`, defeating self-assertion + key-substitution) but not the *governance*; tracked in spirit under backlog C8 (Sigstore + manifest signing, prospective ADR-011, unstarted), but "revocation + attestation expiry + multi-authority scoping are not designed" — net-new specifics.
- **Suggested direction:** Add attestation expiry (re-conform on a cadence / per-version); a revocation list or short-lived attestations + transparency log; support multiple/threshold authorities scoped to *what* they may attest, so no single key is a forge oracle.

### F6 — Credential vending: C5 authorizes the *capability*, not the *mode* — read-only intent is inexpressible (write-escalation)
- **Severity:** Medium
- **Area:** data-plane / contracts
- **Finding:** `vend-credentials` is a single capability URI (`rat://storage/v1/vend-credentials`); the access **mode** (READ vs WRITE) is a caller-chosen *request field* (`VendCredentialsRequest.Mode`, exercised in `composition_storagecreds_test.go:89-90`). C5 authorizes the capability (`registry.Authorize`, `registry.go:67`) and cannot distinguish read from write. So any plugin granted `vend-credentials` for legitimately-read purposes can request `ACCESS_MODE_WRITE` and the gateway allows it (the storage plugin still scopes tenant+prefix, but not the *grant's* intended mode).
- **Why it matters:** A format/engine plugin that only needs to *read* a prefix is, in capability terms, indistinguishable from one that can *overwrite* it. Combined with F1 (the vended cred is then used on the core-bypassing data plane) and review 04's DP-1 (the vended cred is opaque — the core can't constrain it post-vend), a read-scoped grant becomes a write/exfil-and-corrupt primitive. Least privilege is unachievable with the current capability granularity.
- **Freeze-reopen?** no — splitting into `vend-credentials-read` / `vend-credentials-write` capability URIs is additive (new URIs), no message break.
- **Cross-lens:** none yet.
- **Suggested direction:** Either separate read/write capability URIs, or have C5/operator-grant carry the permitted mode and enforce it at the gateway before the provider sees the request.

### F7 — Writable container mounts lack `noexec`; `/data` is world-writable, persistent, and executable
- **Severity:** Low-Medium
- **Area:** core
- **Finding:** The `/tmp` tmpfs is mounted `rw,nosuid,nodev` — **no `noexec`** (`core/deploymentruntime/podman.go:82`). The persistent `/data` mount is created `0o777` (world-writable) and carries no `noexec` either (`:99-107`); the code itself flags 0777 as a shortcut a production runtime would fix. Root fs is correctly read-only.
- **Why it matters:** Read-only-rootfs hardening is meant to stop an attacker from staging executables; leaving the writable mounts executable partially defeats it — a compromised plugin writes a payload to `/tmp` or (persistently!) `/data` and executes it, which aids container-escape staging and survives `Terminate`+relaunch via `/data`. World-writable `/data` is also a defense-in-depth smell even though per-plugin mount-ns isolation prevents cross-plugin reads today.
- **Freeze-reopen?** no — mount-flag/permission changes are runtime-internal.
- **Cross-lens:** none.
- **Suggested direction:** Add `noexec` to `/tmp` and `/data`; map `/data` ownership to the plugin uid instead of 0777 (already noted in-code as the production fix).

## Calibration — what's genuinely strong (don't regress)

- C5 authz **derived** from manifests, audited allow-and-deny (C4), with the terminal stream-close record — `gateway.go:196-288`. Real, not theater.

(The seccomp and resource holes above are the residue, not the whole.)
