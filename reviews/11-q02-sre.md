# Q02 external review (SIMULATED dry-run) — SRE / operability
Reviewer persona: on-call for K8s controller-managers + a Temporal cluster at scale; wrote the etcd-latency-flap and CrashLoopBackOff retros  ·  Lens: sre  ·  Date: 2026-06-02  ·  Provenance: SIMULATED (AI persona)

## Bottom line

**No — I would not carry this pager yet.** Not because the design is wrong (the reconciliation-as-truth + data-plane-bypass choices are genuinely the best calls in the architecture, and sre#4's crash-loop backoff + lease-thrash guard are real and well-built), but because the Phase-1 core's reliability surface has several concrete code-level wedges that the green badge hides, and they sit on the *control-plane survivability* axis that decides whether a 3am incident is a 10-minute fix or a total outage. The single gap to close first — and it is **free to close now while it's still paper** — is the **state-backend survivability seam**: the leader-election lease (`core/lease`) has **no error channel**, so it structurally cannot tell "I lost the lease" from "the backend is briefly unreachable," and the reconciler reads a **fixed** desired set, so the "always re-read state" guarantee everyone is relying on is *unexercised by any code path*. Against a real postgres/Dynamo, the first backend latency blip makes every replica step down though its lease hasn't expired, and recovery bumps the fencing term — a spurious leadership churn on *every* transient blip, not just a pause. Give the lease an error return + a "hold-through-transient-error until genuine expiry" semantic, and document a read-only degraded mode, **before a durable backend binds the lease interface shape** — after that it's a refactor of the one thing that must never wobble. (Maintainer concedes this is net-new and unbacklogged, and correctly narrows the blast radius — see F1.) Everything else (Degraded black-hole, loop serialization, runaway-plugin resource limits, native metrics, DR) is High/Medium and bookable; the lease seam is the keystone.

## Findings

### F1 — The leader-election lease has no error channel: backend-unreachable is indistinguishable from lease-lost
- **Severity:** Critical
- **Area:** core
- **Finding:** `core/lease/lease.go` — `Store.Acquire` (lease.go:36) and `Store.Renew` (lease.go:51) return only `bool`, never a three-state outcome. The `Elector.Step` (lease.go:100-111) treats *any* non-success from `Renew` as "lost the lease" and steps down (`e.leader = false`, lease.go:105), then tries to `Acquire`. The in-memory `Store` is a single mutex (lease.go:22-27) that can only ever fail a renewal for *one* reason: genuine expiry. But a real state-backend (overview.md:206 — postgres / DynamoDB-strong-reads single-key CAS) has a *second* failure mode: the CAS RPC errors/times out (backend slow/unreachable/pool-exhausted). **Crucially (per architect), the frozen wire ALREADY models this correctly:** `state.PutOutcome` is three-state — `COMMITTED` / `CONFLICT` (lost the CAS, deterministic) / `UNKNOWN` ("timeout, partition, dropped response … a lease renewal that returns UNKNOWN cannot be relied on for fencing and the holder must assume it may have lost the lease", state.proto:118-122). So the *contract* is right; the spike's `bool`-only `Store` **under-models a contract that already distinguishes the three cases**, collapsing `UNKNOWN` into the same `false` as `CONFLICT`. This is an impl gap against an adequate wire, not a wire flaw.
- **Why it matters:** 3am, postgres has a 3-second latency spike (vacuum, failover, noisy query — the single most common K8s/etcd outage shape). Every core replica's lease `Renew` RPC fails or times out simultaneously. Each Elector reads `false` → concludes it lost the lease → steps down (clearing its token) → reconcile stops on every replica. When the backend recovers, recovery forces a fresh `Acquire`, which **bumps the fencing term** (lease.go:42-45) — so every transient blip costs a spurious leadership-term change, not just a pause. The TTL-margin thrash guard does **not** save this: it absorbs *late-but-successful* Steps, not *errored* Renew calls — the error is flattened to "lost" before the guard ever sees it. **Blast-radius correction (per maintainer):** because the API / capability-invoke gateway is **active-active** and only the reconcile loop is leader-gated (ADR-002 D5), the wedge is "no convergence during the blip + a spurious term change," *not* "all control calls dead" — the gateway keeps serving. Still a real availability bug: pipelines stop converging on every backend hiccup, and the term churn is exactly the etcd-slow → apiserver-flaps shape the lease package's own doc (lease.go:3-5) cites as its reason to exist, reintroduced one layer down by the missing error type. (Related: leader-election *correctness* itself is pushed to a tier-0 plugin's conformance — overview.md:206, DynamoDB must run strongly-consistent or two replicas both win the CAS — so the core cannot itself guarantee single-leader; the error-channel gap is the operability twin of that delegation.)
- **Freeze-reopen?** no — and pointedly so: the fix needs **no** wire change because `PUT_OUTCOME_UNKNOWN` is already frozen. The `Store`/`Elector` interface is core code. (Fixing it now is cheap; fixing it after a durable backend plugin is written against the `bool`-only shape is a breaking refactor of the most load-bearing primitive — and the three-state outcome it should return already exists in the contract.)
- **Test-gap (verified with maintainer, not asserted):** the sole `leaderloop_test.go` test (`TestLeaderReconcilesFollowerIdleAndFailover`) injects, at its "thrash guard" assertion (lines 49-56), a *delayed-but-successful* renewal (A renews at +25 within the 30s TTL) — only the late-Step case; failover (lines 58-71) is *genuine expiry* (A stops stepping). `lease_test.go:TestFailoverAfterLeaderStops` / `TestThrashGuardRetainsLeadershipUnderLatencySpike` likewise. **No test injects an erroring renewal — and none can, because `lease.go:36/51` return `bool`.** So the `UNKNOWN`/transient-error-vs-lost path has no code to test and no test. The `UNKNOWN` wire path is itself unexercised by any reference (architect's F1 = same conformance-coverage hole: the riskiest paths have the thinnest cross-impl coverage).
- **Cross-lens:** **maintainer CONCEDED** (Q1) — net-new, unbacklogged, "mine to log"; supplied the fencing-term-churn detail + active-active blast-radius narrowing (folded above) + the precise test citations. **architect** confirmed it's "impl against an adequate contract" — file the lease finding as *impl (the Store should surface `UNKNOWN` per state.proto), not wire* — and we cross-reference on the `UNKNOWN`-unexercised coverage hole (his F1). **security** co-files the reconcile-stall consequence (F2).
- **Suggested direction:** make the `Store` surface the frozen three-state `PutOutcome` (or an equivalent `(outcome, err)`), and in `Step`: on `UNKNOWN`/transport-error, **do not clear the token or step down** — retain leadership locally until the lease's own expiry clock elapses (the K8s `renewDeadline < leaseDuration` model), treating fencing-critical writes as unconfirmed until a `COMMITTED` renewal lands; only `CONFLICT` / true local expiry steps down. Add the deterministic test the in-mem store can't host today: "backend returns `UNKNOWN` for N Steps shorter than TTL → leader retains; past TTL → clean single failover." This is the *error-path twin* of the *latency* property sre#4 already built — and the contract is already shaped for it.

### F2 — The reconcile loop serializes all work and all reads under one mutex, with unbounded synchronous RPCs to the runtime
- **Severity:** High
- **Area:** core / operability
- **Finding:** `Reconciler.Reconcile` takes `r.mu` for the *entire* pass over every desired plugin (reconciler.go:97-98), and inside that lock `reconcileOne` makes **synchronous, un-deadlined** RPCs to the deployment-runtime: `status()` → `runtime.Healthcheck` (reconciler.go:159), `launch()` → `runtime.Launch` (reconciler.go:145), `fail()` → `runtime.Terminate` (reconciler.go:170). All three use the loop's `ctx` (threaded from `Loop.Run`, loop.go:42) — which carries **no per-call deadline**. The gateway got C3 deadline-bounding (gateway.go:242-251), but the reconciler's *own* calls to the runtime did not. The same `r.mu` also guards the observability reads `Status()` (reconciler.go:196) and `Endpoint()` (reconciler.go:208).
- **Why it matters:** A single deployment-runtime that hangs on one `Healthcheck` (a wedged podman daemon, a plugin stuck in `D` state, a slow image pull surfacing as a blocked RPC) pins the **single-leader reconcile loop indefinitely**: the lock is held, no other plugin in the desired set gets reconciled, and *every other tenant's* pipelines stop converging. This is strictly worse than the brief's "slow plane consumes loop budget" framing (Q-D/G): it's not a budget problem you can tune, it's an unbounded blocking call under a global lock with no timeout. And because `Status()`/`Endpoint()` share `r.mu`, your **observability goes blind at exactly the moment you need it** — an operator calling `Status()` to ask "why is the loop stuck?" blocks on the same hung RPC. This is the "one slow controller blocks the workqueue" lesson (reviews/03 Concern 8) made concrete in code: no per-item deadline, no separate read path, no fairness.
- **Freeze-reopen?** no — reconciler behaviour, no wire change.
- **Cross-lens:** **security CONCURS and co-files this as a tenancy/availability finding** — "the data-plane bypass does NOT contain it: the bypass keeps *bytes* off the core, but the stall is in the *control* plane (the single-leader reconciler + global mutex) which ALL tenants share, so blast radius is full cross-tenant on convergence/Status reads." We agreed on the shared framing **"availability is a tenancy boundary the spike doesn't isolate"** so the synthesis sees one story; F2 is the control-plane half, F7 the resource half, of the same multi-tenant availability breach. Builds on reviews/03 #8 + sre#4 (which capped *crash-looping* plugins but left *hung-but-not-crashed* calls unbounded).
- **Suggested direction:** wrap every runtime RPC in `context.WithTimeout` (a reconcile-call budget, e.g. < tick interval); snapshot the read fields under a short lock and release before the network calls, or give `Status()`/`Endpoint()` a separate RWMutex / atomic snapshot so reads never block on a hung provider; longer-term, per-pipeline work budget so one stuck plugin can't consume the whole pass.

### F3 — `Degraded` is a terminal black hole: no recovery API, no auto-retry, no emitted signal
- **Severity:** High
- **Area:** core / operability
- **Finding:** When a plugin hits `CrashLoopCap` consecutive failures, `fail()` sets `state = Degraded` (reconciler.go:174-176) and `reconcileOne` thereafter returns immediately and forever (`case Degraded: return`, reconciler.go:112-113). There is **no `Reset()` / `Resume()` / `Uncordon()` method anywhere in `core/`** (grep-confirmed), so nothing can move a plugin out of `Degraded` except restarting the whole core process. The transition emits **nothing** — no event on the bus, no metric, no log hook; `Status()` (reconciler.go:195) surfaces it only to an in-process caller that thinks to poll.
- **Why it matters:** The cap correctly stops a crash-looping plugin from hammering the runtime (sre#4 — good). But the *recovery* side is a permanent dead-end. 3am scenario: an engine plugin crash-loops because *its* upstream (a DB, a credential vendor) was briefly down. It blows the cap in ~30s and goes `Degraded`. The dependency recovers two minutes later — but the plugin **stays dead until a human restarts the core**, and *nothing paged*, because no signal was emitted. The operator's first symptom is a user complaint hours later: "my pipeline silently stopped running." This diverges from the K8s lesson it cites: CrashLoopBackOff *never gives up permanently* — it caps the backoff interval and keeps retrying, so transient faults self-heal. RAT's permanent-Degraded trades self-healing for a guaranteed manual intervention with no alert. F3 compounds F2's blindness: the one state an operator most needs to see is the one the core most quietly hides.
- **Freeze-reopen?** no — reconciler behaviour + an additive event/metric.
- **Test-gap (verified):** `reconciler_test.go:118-124` (`TestCrashLoopBackoffCapAndNoHammer`) *asserts* "Degraded → no more activity, ever" — so permanent-terminal-Degraded is **codified as the tested contract**, not a coverage oversight. `TestRecoveryResetsBackoff` (line 177) proves recovery only from *Pending* backoff (a plugin that goes healthy *before* the cap); there is **no test of recovery from Degraded**, because there is no code path to it. The green suite pins the black-hole shut.
- **Cross-lens:** **maintainer CONCEDED** (Q2) — "terminal-Degraded was a spike simplification, not a defended decision; the right shape is capped-infinite-retry (cap the *interval* at MaxBackoff, keep retrying) + emit a transition event/metric, with Degraded reserved (if kept) for truly-unrecoverable cases (I9 refusal / bad image) and *still* paged." Partial backlog coverage only: sre#8 would carry the metric, but the terminal-vs-capped decision + the resume path + the on-transition event are **net-new / untracked**. Independent of sre#4 (which validated the *backoff*, not the *terminal* semantics or the missing signal).
- **Suggested direction:** keep the cap, but (a) cap the *backoff interval*, not the *attempts* — retry forever at `MaxBackoff` (with jitter) so transient faults self-heal, OR add a `Reset(name)`/`Resume(name)` API + a periodic re-probe of `Degraded` plugins; and (b) **emit a state-transition event + a metric** on every `→ Degraded` / `→ Healthy` edge so it's alertable. A silent terminal state is the worst of both worlds.

### F4 — The tier-0 state-backend SPOF has no degraded mode, and the "always re-read state" guarantee is unexercised (read path unbuilt)
- **Severity:** High
- **Area:** premise / core
- **Finding:** The docs are now honest that state-backend + deployment-runtime + bus are tier-0 bootstrap plugins (overview.md:61; plugin-architecture.md "hidden tier 0") — good, that closes reviews/03's "the framing hides it." But the *operability* of that SPOF is still unaddressed in design. Two concrete gaps: (1) there is **no degraded/read-only mode** — overview.md:218 says "the core stays stateless (state in state-backend plugin)," which means state-backend down ⇒ no desired-state reads *and* no lease (F1) ⇒ total wedge, with no documented "serve last-known-good / read-only" fallback. (2) The Phase-1 reconciler reads a **fixed** desired set passed to `New` (reconciler.go:82-83: *"The desired set is fixed for the spike (the full core reads it from the state-backend)"*) — so the level-triggered "each pass re-observes; events are hints; the reconciler always re-reads state" guarantee (reconciler.go:4-5) is **demonstrated only against an immutable in-memory slice**. The actual state-backend read path — the thing whose latency/availability profile determines every 3am scenario — does not exist in the sealed core. The guarantee everyone leans on is asserted, not exercised.
- **Why it matters:** This is the etcd-is-the-SPOF-and-everyone-pretends-otherwise pattern (reviews/03 §K8s comparison) — except here the docs *don't* pretend (credit due: tier-0 is named honestly in plugin-architecture.md, overview.md:61, and ADR-016 §2's bootstrap-seat chicken-and-egg). The gap is not denial, it's that **the mitigation is thin**: "Tier-0 = its HA is the platform's HA" is an honest *label* but not an *operability answer* — it tells the operator "make postgres HA" and stops, offering no in-core behaviour for the (inevitable) window where the backend is degraded. And because the read path is unbuilt, the most important reliability property — that a sick backend degrades the core to "slow/stale," not "wrong/wedged" — has never actually been run against a backend that can be slow or sick. The whole survivability story rests on a code path that isn't there yet. (Scoped by the F1 correction: a degraded backend wedges *convergence*, not the active-active gateway — but for a control plane, "nothing converges and every blip churns leadership" is the incident.)
- **Freeze-reopen?** no — design/impl; the state proto is already frozen with linearizable-CAS + ordered-Watch gates (overview.md:206).
- **Cross-lens:** **architect pushed back, and is partly right:** surviving a *sustained* state-backend outage is **structurally impossible, not a missing mode** — "the registry can't index without the state-backend (it's tier-0 *because* of that chicken-and-egg), so state-backend-down = control-plane-down is structural; these are plugins in the *swap-the-impl* sense, not the *survive-without-one* sense." I concede that: F4 is **not** "add a mode that survives a dead backend." What remains real and narrower: (a) a *transient* blip shouldn't wedge convergence or churn leadership — that's F1's last-known-good-until-expiry, achievable; (b) the "always re-read state" reliability property is unexercised because the read path is unbuilt — still a genuine "the survivability story rests on code that isn't there" gap. **maintainer:** file it as "tier-0 SPOF is real, docs admit it, mitigation thin," not "they pretend." On bootstrap: the core holds a *minimal in-core seat* to bring up the first deployment-runtime, then everything launches through it (ADR-016 §2, temptation ledger stays 0) — the *launch* leg is specified; the **recovery leg** (the seat's own crash/restart + re-attach-vs-relaunch of already-running plugins after a core restart) is the under-specified 3am corner I flagged back to architect. Builds on reviews/01 Finding 6 + reviews/03 Concern 6 / failure-mode #1.
- **Suggested direction:** before the durable backend lands, decide and document the degraded mode: can the leader serve reads from a last-known-good cache and refuse only *writes* when the backend is unreachable (so in-flight convergence holds rather than wedging)? Pair with F1's hold-through-transient-error. Then build the state-backend read path and re-run the "bus/backend sick → slow not wrong" assertion against a backend that can actually be made sick (fault injection), not an immutable slice.

### F5 — Native observability is paper, and the core emits nothing on the transitions that matter
- **Severity:** Medium-High
- **Area:** operability
- **Finding:** Native core `/metrics` + OTel + an SLO doc are acknowledged-but-unbuilt (sre#8, board/sre.md #8, backlog). Confirmed in code: the reconciler's only telemetry is an in-memory `Status()` hook (reconciler.go:191-203) — no endpoint, no exporter, no golden-signal set. Beyond "it's not built yet," the sharper point: the reconciler emits **no event or metric on any state transition** (the `→ Degraded`, `→ Healthy`, `→ Pending` edges in `reconcileOne`/`fail`). So even once a `/metrics` endpoint is bolted on, the data it would need (transition counts, time-in-Degraded, per-plugin failure attempts over time, lease-term churn) isn't being produced at the transition points — it can only be *polled* as instantaneous gauges, which misses fast flaps entirely.
- **Why it matters:** "Why is my pipeline stuck?" is unanswerable from outside the process today. With observability-as-a-plugin, the core's own SLIs (reconcile staleness, lease churn, per-plugin RPC error rate, bus lag) exist only if someone installs a plugin — and the most diagnostic data (state-transition edges) isn't emitted at all. This is the Airflow "scheduler is up but the task is stuck and the logs say nothing" shape (reviews/03 §Airflow), across N polyglot processes. It compounds F2 (reads block on the loop) and F3 (the Degraded edge is silent): the three together mean a stuck pipeline is invisible *and* the one tool to inspect it can hang.
- **Freeze-reopen?** no — additive (a metrics endpoint + emitting on transitions).
- **Cross-lens:** extends sre#8 / board/sre #8 with the fresh "nothing is emitted at the transition" evidence. The brief asks priority/sufficiency — my read: ship the minimal golden-signal set (reconcile staleness p99, lease-term churn, per-plugin transition counts + time-in-Degraded, per-plugin RPC error/latency) as the **prerequisite for F3's alerting**, before multi-tenant.
- **Suggested direction:** emit a counter/event on every reconcile state-transition edge now (cheap, and it's what F3's alert keys off); pin the `/metrics` golden-signal list + an SLO doc while the core is still paper (free to specify, expensive to retrofit consistency).

### F7 — The "real enforcing" podman runtime enforces isolation but NO resource limits — one plugin can OOM/fork-bomb the shared host
- **Severity:** High
- **Area:** core / operability
- **Finding:** The podman deployment-runtime — the *real, kernel-enforcing* runtime that retired the reviews/08 D1 honesty gap (`core/deploymentruntime/podman.go`) — sets the full isolation profile: `--cap-drop=ALL` (podman.go:76), `--read-only` root (podman.go:78), seccomp (podman.go:109-112), non-root, metadata-egress block. But `Podman.Launch` (podman.go:73-93) **never reads `spec.GetLimits()`/`GetRequests()`** and emits **no `--memory`, no `--cpus`, no `--pids-limit`** (grep-confirmed); `checkI9Minimum` gates the *isolation* profile only. The frozen `LaunchSpec` carries `requests`+`limits` as `ResourceQuantities{cpu, memory}` (deployment_runtime.proto:47-49, 68-69) — a *mandatory* manifest field (board/sre.md #C4) — so **the wire has the limits and the enforcing runtime silently drops them.** And it's both runtimes: `LocalProcess.Launch` (localprocess.go:47) doesn't map them either (grep-confirmed). So the security profile is hardened while the *resource* profile is wide open, on every shipped runtime.
- **Why it matters:** Isolation ≠ resource containment. A buggy or hostile plugin that fork-bombs, leaks memory, or pins CPU is *not* crashing — so the reconciler's crash-loop cap (sre#4) and Degraded path never trigger; it just sits there consuming the host. On a shared multi-tenant box that's a direct cross-tenant **availability** hit: tenant A's runaway engine OOM-kills tenant B's plugins (or the core itself) via the kernel OOM-killer, or exhausts PIDs so nothing new launches. This is the K8s "shipped without limits/QoS, spent years on OOM-killer tuning + eviction ladders" lesson (reviews/03 Concern 8), and it lands *behind* the green "real enforcing runtime" badge — an operator reasonably assumes the hardened runtime also caps resources, and it doesn't. It also compounds F2: a host under memory pressure produces slow Healthcheck RPCs, which pin the loop.
- **Freeze-reopen?** no — runtime impl maps already-frozen `resources{requests,limits}` fields to `--memory`/`--cpus`/`--pids-limit`.
- **Cross-lens:** **security raised this** (runaway-plugin DoS; they lean Critical) and I confirmed the missing flags; **maintainer co-CONCEDED it as High net-new**, adding the precise note that **`--pids-limit` absence specifically defeats fork-bomb containment even with cap-drop + non-root** (a fork bomb needs no privilege and writes nothing), and confirming both runtimes drop the frozen field → "the real enforcing runtime enforces *security* isolation but not *resource* isolation." Security frames F7 + F2 as the two halves of one breach: **"availability is a tenancy boundary the spike doesn't isolate."** Separately, security's **ticket-replay** finding (`arrowticket.go:54` in-process per-Minter `used map[string]bool` → a producer restart / 2nd replica reopens the single-use window) is theirs to own; I co-sign the operability half — because the bytes leg bypasses the core, a replayed `DoGet` leaves **no central audit record**, so it's *undetectable* from the control plane (security folded that framing into their F2).
- **Suggested direction:** map `resources.limits` → `--memory`/`--cpus`/`--pids-limit` (the field is already frozen — pure impl) on both runtimes, and reject a launch whose manifest omits limits in multi-tenant mode; add a conformance vector "limit set → exceeding it is contained, not host-wide." Treat it as a multi-tenant gate, not a GA nicety.

### F8 — The deployment-runtime contract has no re-attach verb: a core restart orphans or double-launches the running fleet
- **Severity:** Medium-High
- **Area:** contracts / operability
- **Finding:** `DeploymentRuntimeService` (deployment_runtime.proto:28-43) has exactly three verbs — `Launch`, `Terminate`, `Healthcheck` — and **no `List` / `Adopt` / `Discover`** (grep-confirmed). `Terminate` and `Healthcheck` both key on `instance_id` (proto:91, 107), a runtime-assigned **opaque handle returned only in the `Launch` response** (proto:83-84) and held **in core memory** (`supervisor.go` `instances []string`; reconciler `pluginStatus.instanceID`). `LaunchRequest` (proto:75-80) carries `plugin_id` + `spec` but **no idempotency key** — so `Launch` is not idempotent: two calls for the same `plugin_id` yield two instances with two different `instance_id`s. There is no verb to *enumerate* running instances or to *adopt* one whose handle the core no longer holds.
- **Why it matters:** The whole HA story is "stateless core, N replicas, restart/failover freely" (overview.md:218; reviews/03 calls failover ~15-20s a strength). But on a core restart or leader failover, the in-memory `instance_id` map is gone, and the contract offers **no way to rediscover the plugin processes that are still running** (the data plane is independent, so they *are* still up). The new core can only: (a) do nothing → the running instances are **orphans** it can neither Healthcheck nor Terminate (a leak that accrues across every restart), or (b) re-`Launch` the desired set → **double-launch** (two engine processes, port/endpoint collisions, or — worst — two writers to the same target). Because `Launch` has no idempotency key, even a crash *between* `Launch` returning and the core persisting its `instance_id` orphans that instance with no handle to ever find it again. This is the routine case (every deploy, every OOM-restart of a core replica), not an exotic one — and unlike the other findings it touches the **frozen wire**.
- **Freeze-reopen?** no, but **additive-now**: the fix is a new RPC (`ListInstances` / `AdoptInstance`) + an optional idempotency key on `LaunchRequest` — backward-compatible, so it doesn't reopen `v1`, but it is the one finding here that's a *wire addition* rather than pure impl, so it's cheapest to land while the freeze is still local (the reviews/08 "absorb additives before publish" window).
- **Cross-lens:** **architect surfaced and co-owns this** (F4 recovery-leg → its own seam): "the deployment-runtime proto has no 'adopt an already-running instance' verb that re-attach would need — worth checking whether that's frozen-additive or missing." Verified missing; additive to fix. Distinct from the bootstrap-*launch* leg (solved by the in-core seat, ADR-016 §2) — this is the bootstrap-*recovery* leg.
