## Findings

### F1 — The keystone (context envelope + gateway stamping) is the least-tested part of the most-irreversible wire
- **Severity:** High
- **Area:** contracts
- **Finding:** `common/v1/context.proto` (the three-principal identity model, trace separation, `SubjectAssertion` signing/anti-replay) and the gateway's read/validate/stamp contract are the load-bearing carrier for C1/C2/C3/C5/C7/C8. ADR-009 lists `context.proto` and `core/v1/invoke.proto` among the frozen-and-"exercised" data-plane surface (009 §"cross-cutting wire types"). But ADR-007 §Consequences-Neutral concedes the golden vectors are *unaffected* by the carriage move because "they assert on data behavior, not context carriage" (007:80), and the only thing that exercises the stamping rules is the single stub gateway (`plugins/format/inmemory-go/gateway_test.go`, the ADR-007 surfacing artifact). So the ADR-003 rigor the brief leans on as the safety net — *two technologically-divergent references cross-run on shared golden vectors* — was applied to format/catalog/engine data behavior but **not** to the envelope every cross-cutting property rests on.
- **Why it matters:** The envelope is the part you can least afford to get wrong (identity, tenant, signing, per-hop re-derivation) and the part a single implementation is most likely to get subtly self-consistent-but-wrong. A second independent gateway is exactly what would surface, e.g., an under-specified canonicalization of `rat-callmeta-bin`, a disagreement on which fields are re-stamped vs propagated on a stream-open vs per-frame, or the M4 bare-mirror cross-check being honored by one impl and skipped by another. Discovered post-publication, any of these is a v2 on the universal carrier — the single worst place to take a breaking change.
- **Freeze-reopen?** No (no field changes proposed) — but it *weakens the freeze-confidence claim* the brief makes; I would not treat the context wire as "tested-by-2-impls" the way the data axes are.
- **Cross-lens:** **Maintainer CONCEDED** — "the riskiest frozen surface has the weakest conformance … exercised by exactly one implementation (the spike's Go gateway + gateway_test.go + composition), no portable conformance vector, no independent second-impl cross-run; 'tested by ≥2 impls' does NOT extend to it" (gateway stamping at gateway.go:196-253). Aligns with the un-dissolved board caution reviews/09 ("green certifies shapes, not obligations"). **Security RESOLVED the escalation question:** the *control-plane* carriage (C2 swaps the identity source, gateway.go:204; wire unchanged; provider already authenticates the core peer) is an additive **patch, not v2** — so F1 stays freeze-reopen NO. The bytes-leg half of the same property is a *separate* finding (see F9).
- **Suggested direction:** Add a context-carriage conformance suite (vectors that assert: missing-traceparent → reject; `caller_plugin` re-derived not propagated; M4 cross-check enforced; stream open-stamp semantics) and require two independent gateway implementations to cross-pass it *before* the envelope is treated as frozen-with-confidence. This is the ADR-003 forcing function applied to the one axis-independent contract it skipped.

### F2 — Method-level capability URIs don't negotiate *within-method* semantic evolution — the silent-additive-field trap
- **Severity:** High
- **Area:** contracts
- **Finding:** The frozen `capabilityRef` is a bare URI, `{capability: "rat://axis/v1/x"}`, major-only, no ranges, no attributes (`plugin.v1.json` $defs.capabilityRef, lines 179–186; `additionalProperties:false`). The project's answer to rev01-F4 ("major-only forces flag days") is *capability-per-method granularity*: a new feature is a new capability URI you can `require` exactly (e.g. catalog gained `register-table`/`commit-table` additively, ADR-010). That works for new *methods*. It does **not** work for behavior added *inside an existing method* as a new field: `WriteResult.already_applied` (data.proto:98–103), `WriteResult.snapshot_id` presence (88–97), `ArrowStream.expected_rows`/`expected_batches` (79–82). A consumer that *relies* on `already_applied` to make a replay a no-op has no way to express "I require a provider that honors idempotency" — proto3 makes the field silently absent (=`false`) on an old provider, so the consumer treats a previously-committed write as a fresh one.
- **Why it matters:** This is rev01-F4's rigidity re-entering through the door the per-method-capability design left open. The platform's whole pitch is mix-and-match plugins of independent vintages; the moment a method's *contract semantics* (not just its shape) evolve additively, there is no negotiation handle, and the failure is silent data-correctness drift, not a clean error. The only in-model escapes are both bad: (a) mint a new capability URI per behavioral variant (`rat://format/v1/merge-idempotent`), proliferating the capability space and fragmenting the ecosystem; or (b) a v2. The frozen wire has no `min_capability_revision`-style discriminator to avoid either.
- **Freeze-reopen?** No (it's a consequence of the chosen model, not a broken field) — but it is the structural seam most likely to *force* a future capability-URI proliferation or v2, so it belongs on the v2-design radar now.
- **Cross-lens:** **Maintainer CONCEDED, net-new.** The contract *does* support the escape (a distinct `rat://format/v1/idempotent-write` capability the requirer `requires` + C5 enforces) — "but that was **not applied to the ADR-012 fields**; they shipped as plain additive fields, trusting both sides upgrade together. So today there is no skew protection for additive *semantic* fields within a major … it should be a new capability URI per behavior, the contract supports it, but we didn't do it for the one case (crash-safety) where it bites hardest — so it's an **undefended seam, not a decided v2**." ADR-012's "demonstrated in composition" proves both-new, never new-vs-old. Maintainer weights this as one of the two most important of my three questions (silently breaks the headline idempotency guarantee under skew).
- **Suggested direction:** Decide the policy *now* and write it down: either (i) a convention that any behavior-bearing additive field gets a paired capability URI (and document the ceiling on URI proliferation), or (ii) add — additively, while you still can — an optional `requires[].min_revision` integer that the gateway checks against a provider-declared capability revision, giving graceful negotiation without SemVer ranges. Option (ii) is the npm-peerDependencies lesson rev00 cited as a *strength* (synthesis "what's strong" #8) but only bought at the major level.

### F3 — The freeze gate is structurally blind to *omission*; ADR-010/B1 is the existence proof, not the last case
- **Severity:** High
- **Area:** contracts
- **Finding:** ADR-010 records that catalog froze (ADR-009, `rat/1`) functionally incomplete: the create→write→register→merge loop had no wire path for its middle, and the cross-axis composition *passed anyway* only because the harness "reach[ed] into the catalog's private store … the green badge is propped up by a stand-in the contract can't express" (ADR-010 §Context). Two independent references and a golden-vector cross-run did not catch it; a *board review* (reviews/08 B1) did, post-freeze. The reason is structural: golden vectors assert on the behavior of capabilities that **exist**. They cannot assert on a verb that is **absent** — there's no vector for a method nobody wrote. The two-reference gate (ADR-003) is a defense against *wrong* contracts, not *incomplete* ones, and when both references stub the same gap the same way (both in-memory catalogs seeded out-of-band), they agree their way past it.
- **Why it matters:** ADR-010 was patchable additively — good, and it validates the additive door. But B1 was HIGH and "three specialists converged on it" only *after* the freeze tag. The architect's scar here: the process the brief trusts to have caught the obvious blockers has a known blind spot for *missing* capability, and B1 is unlikely to be the only one (catalog's own Q02/Q03 in ADR-010 — per-table history, merge-consuming-commits — are the same shape of "the references demonstrate the wire faithfully but a real backend exercises corners the vectors don't"). The risk isn't a wrong field; it's discovering at month 9, against a real Iceberg/Nessie backend, that another headline loop has an out-of-band stand-in.
- **Freeze-reopen?** No (the fixes are additive by construction) — but it should temper "the obvious blockers were caught" into "the obvious blockers *that exist as vectors* were caught."
- **Cross-lens:** Confirms rev01-F8 ("contracts frozen before plugins stress them") with fresh post-freeze evidence; the additive door (ADR-009 R3 → ADR-010) is the mitigation working, not the absence of the risk.
- **Suggested direction:** Before each *real-backend* reference lands (Iceberg format, Nessie catalog, postgres state), run an explicit "what loop can this backend complete that the in-memory refs faked?" omission-audit — make finding the next B1 a gated step, not a board-review surprise.

### F4 — "The core does six things" is an unfalsifiable metric; the enforcement layer is the real, unbudgeted core
- **Severity:** Medium-High
- **Area:** premise
- **Finding:** The discipline metric is "the core stays six things" guarded by a temptation ledger (currently 0). But the cross-cutting properties C1–C10 are explicitly defined as "*correctness conditions of the existing six*, not new core responsibilities" (plugin-architecture.md §Cross-plugin concerns; ADR-005 §Positive "absorbs this without a 7th thing"). That framing makes the metric un-trippable: the invoke gateway already performs, on a single hop, C2 (authn) + C5 (capability authz) + C7 (tenant scoping) + C3 (state-namespace derivation) + C8 (audit emission) + C1 (trace validate/stamp) + deadline bounding + routing (invoke.proto:14–31) — six-to-eight distinct enforcement jobs — and every one of them was added while "the core stays six" remained literally true. The ledger counts new *kinds*; it cannot count enforcement-layer accretion, which is where the real complexity (and the real audit surface, and the real bugs) live.
- **Why it matters:** rev01-F6 said the count undercounts to ~9 (tier-0). This is the other half: the count can't *grow*, so unbounded policy/enforcement complexity hides inside "the existing six" with no budget and no drift signal. The thing you're protecting — a small, auditable core — is measured by a number that stays 6 whether the enforcement layer is 2k or 20k LOC. K8s lived this exactly: the apiserver "stayed one thing" while admission control, authz webhooks, and conversion accreted into the most complex, most security-sensitive code in the system. The metric should track the surface that actually carries the risk.
- **Freeze-reopen?** No — it's a discipline/observability issue, not a wire issue. But it's a premise-level "wrong line in the core boundary" the brief asked for: the boundary is drawn around six *nouns* when the risk lives in the *verbs the gateway enforces*.
- **Cross-lens:** Builds on rev00 Theme 1 ("everything is a plugin has a load-bearing blind spot for cross-cutting concerns") and rev01-F6; net-new is the *metric-unfalsifiability* angle.
- **Suggested direction:** Add an explicit "enforcement-layer LOC + obligation count" line to the temptation ledger, with its own threshold. When the gateway gains its Nth enforced obligation, that should trip the same "is this really a correctness condition of an existing thing, or a new thing in disguise?" review the 7th-noun rule triggers.

### F5 — Tenancy: the structural primitives are flat-string equality; the sharing/hierarchy decisions the axis exists to make have no wire to be *actioned*
- **Severity:** Medium-High
- **Area:** data-plane / contracts
- **Finding:** C7 made tenancy structural (good, answers rev01-F3): `Identity.tenant` threads every hop (context.proto:117–123), the state gateway namespaces by `(caller_plugin, tenant)` (state.proto:14–21), storage vends tenant-scoped creds, and "the core refuses cross-tenant access" (context.proto:120). All of this is **flat-string equality**. Yet the tenancy axis ships `DECISION_KIND_SHARING` and `counterparty_tenant` (tenancy.proto:39–53) — it is explicitly built to answer "may tenant A share dataset X with tenant B?" There is **no frozen wire shape** by which an *allowed* SHARING decision is enforced: state keys are namespaced by exact tenant with no delegation/grant primitive, and `storage.VendCredentials` is tenant-scoped with no "vend for counterparty under grant G" shape. The vision sells a "hierarchical tenancy plugin" (overview.md:81, vision.md:29 "compliance is a configuration"); the frozen structural layer can express *isolation* but not *controlled crossing*.
- **Why it matters:** The decision is computable and un-actionable — the tenancy plugin can return `allowed=true` for a cross-tenant share and nothing in the frozen primitives can carry that grant into the state namespace or the vended credential. So either (a) hierarchical/sharing tenancy is effectively a v2 (the structural primitives need a delegation/grant dimension), or (b) every sharing implementation smuggles hierarchy into the opaque `tenant` string and reimplements crossing out-of-band — which is precisely the "isolation is an emergent property of N plugins agreeing" failure C7 was created to kill, reintroduced one layer up. This is a load-bearing concern that is *neither* cleanly one of the six *nor* cleanly a plugin (brief question A): the crossing rule must be core-enforced (like isolation) but is shaped by a plugin (like policy), and the frozen wire only built the isolation half.
- **Freeze-reopen?** Yes — real hierarchical/sharing tenancy needs a delegation/grant shape in the `state` and `storage` wires (or in the identity envelope), and adding the *semantics* of cross-tenant access to flat-equality primitives is not obviously additive.
- **Cross-lens:** **Maintainer CONCEDED** — grepped and confirmed "no delegated-tenant primitive exists in state/storage … the SHARING decision is currently **advisory at the policy layer with no downstream consumer**. Honest answer: yes — actioned cross-tenant sharing and hierarchical tenancy are effectively v2. v1 delivers flat structural isolation + a sharing decision hook that nothing actions." Net-new as a stated gap (R1 covers a different confused-deputy concern). Inverse of rev01-F3 / rev00-C7: they fixed "isolation emergent"; this is "sharing un-actionable."
- **Suggested direction:** Decide whether v1 tenancy is *isolation-only* (then say so — drop or mark `DECISION_KIND_SHARING` as policy-advisory-not-enforced, so the axis doesn't advertise an un-actionable verb) or *sharing-capable* (then the delegation primitive belongs in `rat/1` before publication, because retrofitting cross-tenant semantics onto the state namespace is the expensive kind of v2).

### F6 — Capability negotiation resolves *what*, never *which*; provider selection falls to a desired-state language that isn't a frozen artifact
- **Severity:** Medium
- **Area:** premise / contracts
- **Finding:** When two plugins `provide` the same capability, the spike *refuses* the ambiguity (brief question F), and the frozen `provides` entry carries no discriminator to break it — it's a bare URI (`plugin.v1.json` provides → capabilityRef, no name/attributes). So "use Delta for the bronze layer, Iceberg for silver" cannot be expressed by capability negotiation; it must be expressed by operator binding (overview.md "planes: [name, axis-bindings]", reconciliation model). But there is **no frozen schema for planes / pipelines / subscriptions / axis-bindings** — `contracts/schema/` contains only `plugin.v1.json` + per-kind manifest schemas; the reconciler's *input language*, where selection actually happens, is unspecified and unfrozen.
- **Why it matters:** The contract triple (`.proto` + `plugin.yaml` + URIs) is sold as the composition surface, but the actual *which-provider* binding lives in an artifact outside the triple and outside the freeze. That's a real hole in "the contract is the platform's wire shape": a third party can write a perfectly conformant `format` plugin and still have no contracted way for an operator to select it over a peer, because the selection language isn't a contract yet. It also means capability negotiation — billed as "the cleanest single idea in the design" (rev00 strong-#7) — is, by itself, *incomplete* for the headline "mix per layer" use case; it needs an unspecified binding layer to be usable.
- **Freeze-reopen?** No (the plane/binding schema is simply not yet written) — but it should be designed and (when stable) frozen with the same rigor, because it's load-bearing for provider selection and currently invisible.
- **Cross-lens:** **Ecosystem made this their F3 (High), from the adoption side** — attribute-less URI matching makes negotiation *over-promise*: `requires: rat://format/v1/scan` "matches" a format that physically can't read the author's tables (parquet-only vs delta), and the failure surfaces inside the *consuming* plugin's name (the strategy author eats the blame for the format author's incapacity — rev02 Stage-9, now structural). We're aligned: I flag it from the wire-correctness side (selection has no discriminator + plane-binding isn't a frozen artifact), they from the author-blame/adoption side. Ecosystem also finds the *contract triple itself* good ergonomics, so this is narrowly about *selection*, not the whole surface. On the conformance/trust gate (their other worry) I confirmed it is **tiered (soft solo / required-signing team+)** but *designed-but-unbuilt today* (plugin.v1.json:6 "currently self-asserted — no enforcer exists yet") — so selection AND trust-enforcement are both unfrozen/unbuilt while billed as part of the model.
- **Suggested direction:** Adding an optional `attributes`/`selector` to `capabilityRef` is **additive** (existing manifests still validate — not a wire-reopen), but the granularity decision must be made **before third-party plugins exist** — retrofitting attribute-matching across a populated ecosystem is the exact orthogonality-assumption flag-day ADR-003 exists to prevent (ecosystem F3 concurs). Separately, name the desired-state/plane-binding language as a first-class contract artifact, and document that capability negotiation resolves *eligibility* while plane bindings resolve *selection*.

### F7 — Async-plane identity is unsigned in-body through a pluggable transport, while AuditRecord is core-signed — a frozen asymmetry on isolation-bearing data
- **Severity:** Medium
- **Area:** contracts
- **Finding:** ADR-007's signature win — "there is no wire copy in the body" to forge (007 §Positive) — holds only for the sync path. The async `Event` carries `RequestContext context = 1` **in-body** (event.proto:37–46), and `context.identity.tenant` is used for "routing/isolation" (event.proto:23–24, 46). The integrity argument is "core-stamped once at emit … not caller-asserted on the bus" (event.proto:21–24) — but the bus *transport* is a plugin (NATS/Kafka/Redis, ADR-002 D2), so on the async plane the carrier is a third party, and the envelope is **unsigned**: `caller_plugin`/`tenant`/`source` are bare strings (only `SubjectAssertion` is self-verifying). Contrast `AuditRecord` (audit.proto:39–80), which the core **signs + hash-chains** precisely so a sink "rejects any record whose signature does not verify." The most security-sensitive async field got the *weaker* treatment than the audit record.
- **Why it matters:** A compromised or buggy bus-transport plugin — or anyone who can publish to a subject — can inject an `Event` with a forged `context.tenant`, and a subscriber fanning out has no per-hop channel to re-derive it (unlike the sync gateway re-stamp). Tenant-scoped event routing/isolation then trusts unsigned data delivered by a pluggable component. This is the wire-shaped half of synthesis I6 ("bind publisher identity to events"), which was booked as a by-GA *operational* item — but the frozen *shape* already chose unsigned-in-body, and the asymmetry with the signed AuditRecord is the tell that one plane's integrity was hardened and the other's was deferred at the contract level.
- **Freeze-reopen?** No — a signature/`key_id` pair can be added to `Event` additively (same shape as AuditRecord), and SubjectAssertion already self-verifies. Flagging so the additive lands before publication, not after subscribers trust the unsigned field.
- **Cross-lens:** Extends rev00 I6 / rev01-F3 with the fresh frozen-wire evidence (Event unsigned vs AuditRecord signed); defer to security on exploitability.
- **Suggested direction:** Mirror the AuditRecord pattern on `Event` (core signature + `key_id` over canonical bytes) before GA, and until then document that `Event.context` integrity depends on a trusted bus deployment — don't let "core-stamped" read as "tamper-evident."

### F8 — The front-door architecture doc describes a manifest the frozen schema rejects
- **Severity:** Low
- **Area:** contracts
- **Finding:** `overview.md` §"plugin.yaml manifest" (lines 115–143) shows `provides` with `name`/`version`/`schema`, `requires` with `capabilities: [...]` and `version: ">=v1"`, and `suggests`/`contributes` shapes that the **frozen** `plugin.v1.json` does not accept: capabilityRef is `{capability: <uri>}` only (`additionalProperties:false`), there are no version ranges, and `requires` is a flat URI list. The documented manifest would fail validation against the contract it's documenting.
- **Why it matters:** For a project this obsessive about honesty (and whose brief explicitly worries about "badge over-promises reality"), the canonical architecture doc showing a fictional manifest is a credibility paper-cut — a real reviewer or first plugin author will copy it, hit a schema error, and lose trust in the doc set. It also quietly hides F2/F6 (the `version: ">=v1"` and per-`provides` `schema` the example shows are exactly the expressiveness the frozen schema *dropped*).
- **Freeze-reopen?** No — doc fix only.
- **Cross-lens:** none.
- **Suggested direction:** Regenerate the overview.md manifest example from a real `contracts/examples/*.plugin.yaml` (which do validate), and add a one-line note that capability refs are URI-only by design (pointing at the F2 rationale).

### F9 — The bytes leg has no normative producer-channel-auth MUST; the ArrowStream ticket degrades to honor-system header-trust (joint with security)
- **Severity:** Medium-High
- **Area:** data-plane / contracts
- **Finding:** The frozen `ArrowStream.ticket` (data.proto:53-63) is correctly opaque and the *comment* requires it be short-TTL/single-use/bound-to-`{caller_plugin,tenant,stream}` (SEC-14). But the core-mediated identity guarantees (C2/ADR-007) reach only the *control* plane — the bytes leg bypasses the core entirely (overview.md, ADR-005 §exception), so there is no core peer on it. The same integrity argument ("identity rests on transport auth, provider authenticates the core as peer") therefore degrades on the bytes leg to "rests on the *third-party producer's* transport auth," which **no frozen MUST obligates.** Security's read (which I adopt): there's no normative conformance clause requiring an `ArrowStream` producer to verify that the *presenting channel's authenticated identity* equals the ticket-bound `{caller,tenant}`; the reference trusts raw transport/app headers (`bulkleg_test.go:39`). The wire field is fine — the missing thing is a contract-surface obligation.
- **Why it matters:** This is the data-plane half of F1: the bytes path is the one place the core *structurally cannot* mediate, and it carries the actual user data across the tenant boundary. Without a normative producer-channel-auth MUST, a leaked/replayed ticket presented over an unauthenticated channel is the cross-tenant data-exfil path the whole tenancy story is supposed to close — and it's exactly the kind of clause that is expensive to add *after* third-party producers ship trusting headers.
- **Freeze-reopen?** Yes (soft) — no message break, but a normative frozen conformance MUST ("a conformant ArrowStream producer MUST verify presenting-channel identity == ticket-bound {caller,tenant}; transport/app headers are insufficient") is best added to the frozen contract surface *pre-publication*. Primarily security's F1; recorded here because it's contract-shaped.
- **Cross-lens:** **Security owns this** and split my C2 question to surface it: control-plane C2 = additive patch (F1 stays freeze-reopen no); bytes-leg = this. Concur fully.
- **Suggested direction:** Add the producer-channel-auth MUST to `data.proto`'s `ArrowStream` prose + a conformance vector that fails a ticket presented over a channel whose authenticated identity ≠ the ticket binding, before any real (polyglot, third-party) producer reference ships.
